# 🎯 Vibe Coding 案例：用 AI 從零打造並上線 PredictCup 2026

> 一個世界盃預測 Web App，從**一份企劃書**到**可部署上線、能分享給朋友**的全端系統。
> 全程以「人出方向與決策、AI 出架構與實作」的協作模式完成。本文整理開發歷程、
> 人機分工、踩過的雷與沉澱出的規則，供想用 AI 做真實專案的人參考。

---

| 項目 | 內容 |
|------|------|

## 2. 開發歷程（時間軸）

開發**嚴格分階段**，每階段做完先驗證、再決定下一步——這是品質的關鍵。

| 階段 | 內容 | 產出 |
|------|------|------|
| **Phase 1** | 後端架構與 API 設計 | SQLAlchemy 5 張表、RESTful 路由表、結算規則文件 |
| **Phase 2** | 結算核心邏輯 | 純運算結算引擎 + 投注/排行榜服務、14 個單元測試 |
| **Phase 3** | 前端 Layout + 首頁 | Next.js 架構、底部導覽、Home Dashboard（Mock）|
| **Phase 4** | 前後端串接 | 賽事大廳、下注 Modal、排行榜、私房聯賽、個人中心 |
| **Phase 5** | SSO/AI/Cron/部署 | JWT+Google SSO、Claude 預測、自動結算、Docker |
| **真實化** | 真實賽程 + 後台 | 上網抓 2026 世界盃 72 場、「更新賽果」後台、可重入結算 |
| **管理者** | 角色權限 | Mike 設為管理者、後台鎖權限 |
| 🔴 **上線強化** | 阻擋性安全 | 關 dev 登入、密鑰 fail-fast、CORS、DB migration、排程鎖 |
| 🟠 **上線強化** | 穩定性 | 併發鎖、限流、Sentry、readiness、稽核、R32、CI/CD |
| **維運** | 文件 + 部署 | 上線手冊、Railway 清單、推上 GitHub |

> 💡 **關鍵心法**：不要一次叫 AI「把整個 App 做完」。**切成可驗證的小階段**，
> 每段都能跑、能看、能測，錯誤才不會累積到難以收拾。

## 3. 人與 AI 的分工

Vibe Coding 不是「AI 全包」，而是**各司其職**。這個專案的分工：

### 👤 人（產品/決策者）
- **提需求與企劃**：一份 PDF/Word 企劃書定義產品。
- **做商業決策**：道具能否同時用？扣點在下注時還是結算時？賽程用即時 API 還是靜態真實資料？這些 AI 會主動詢問，由人拍板。
- **控節奏**：要求「只先做 Phase 1，做完問我」，避免 AI 暴衝。
- **決定優先序**：「先做 🔴 那批」「A、B 都做」。
- **提供認證與資源**：Google OAuth、GitHub 帳號登入、API 金鑰——**這些 AI 不能也不該代勞**。

### 🤖 AI（架構/實作者）
- **架構設計**：分層（純運算 vs 副作用）、資料模型、API 設計。
- **寫程式 + 寫測試**：功能與測試一起產出，保持全綠。
- **自我驗證**：跑測試、build、開預覽伺服器**實機截圖**確認畫面，而非嘴上說「應該可以」。
- **主動標風險**：發現安全漏洞/限制就直說（見第 5 節）。
- **寫文件 + 處理 git**：README、部署手冊、commit、推送、清歷史大檔。

> 一句話：**人負責「做什麼、為什麼」，AI 負責「怎麼做、做好驗證」。**

## 4. 讓 Vibe Coding 成功的關鍵做法

1. **分階段 + 階段閘門**：每階段結束「暫停等指示」，人有機會校正方向。
2. **每一步都驗證**：
   - 後端：`pytest`（純結算邏輯免相依也能跑）
   - 前端：`tsc --noEmit` + `next build`
   - 整合：開 dev server **截圖**看真實畫面、用 curl 跑 API
3. **誠實回報**：測試失敗就貼失敗、有限制就講限制，不粉飾。
4. **決策交還給人**：遇到商業/不可逆選擇（資料源、repo 公開與否）用選項詢問，不擅自決定。
5. **文件即交付物**：邊做邊寫 README/DEPLOY/GO_LIVE，讓成果可被別人接手與部署。

## 5. 開發應注意事項（真實踩過的雷）

這些都是這個專案**實際發生**的狀況，特別值得記下來：

| 雷 | 狀況 | 教訓 |
|----|------|------|
| **安全預設值** | `ALLOW_DEV_LOGIN=true` 讓「任何人輸入 Mike 就變管理者」 | 正式環境的危險預設要**主動關閉並 fail-fast** |
| **金鑰外洩面** | 前端 `NEXT_PUBLIC_*` 會被打包進瀏覽器；金鑰絕不能放 | 金鑰只在後端、只在平台 Secret，永不進 repo/前端/映像 |
| **大檔進 git 歷史** | 109MB 的 zip 混進某個 commit，GitHub 擋推 | 早早 `.gitignore` 產物；進歷史就得 `filter-branch` 重寫 |
| **環境差異** | 容器平台用 `$PORT` 注入埠號，寫死 5001 會起不來 | 綁 `${PORT:-5001}`；別假設本機環境＝雲端 |
| **AI 的知識邊界** | AI 訓練資料到 1 月，**不知道 6 月實際賽果** | 即時資料**必須上網查證**，不能讓 AI 編造 |
| **第三方資料源** | API-Football 免費版沒有 WC2026；維基爬蟲會隨改版壞 | 設計**可插拔來源 + 後援**，並誠實標註脆弱性 |
| **並發安全** | 多 worker 排程器會重複結算；併發下注會重複扣道具 | Redis 分散式鎖 + DB row lock + **結算可重入** |
| **工具互打架** | `npm run build` 會清掉 dev server 的 `.next` | 別在 dev server 跑著時做 production build |
| **相對路徑陷阱** | `sqlite:///dev.db` 在 CLI 下行為不穩 | 關鍵流程用絕對路徑，消除歧義 |

## 6. 沉澱出的程式規則（Coding Rules）

這個專案逐步建立、值得套用到其他專案的規範：

**架構**
- **純運算與副作用分離**：`scoring.py`（無 DB、可單測）vs `settlement.py`（寫 DB）。
- **設定集中**：所有環境變數/業務常數進 `config.py` / `constants.py`，單一真實來源。
- **用 Enum 約束狀態**：賽事狀態、階段、下注選項都用 enum，避免髒資料。
- **可插拔來源**：賽果同步 = API-Football 或維基，由設定決定，介面一致。

**正確性與安全**
- **結算可重入（idempotent）**：重抓比分先反轉舊分再套新分，怎麼跑都對。
- **正式環境 fail-fast**：密鑰/CORS/Google 沒設好就**拒絕啟動**並印原因。
- **權限以裝飾器集中**：`@admin_required` / `@login_required`，不散落判斷。
- **稽核留痕**：管理者操作寫 `AuditLog`。

**流程**
- **測試保持全綠**，新功能必配測試（含權限被拒、可重入等邊界）。
- **migration 取代 `create_all`**：schema 變更可控、可回滾。
- **commit 訊息結構化**：標題 + 分類條列 + 共同作者；**不 commit 產物/金鑰**。
- **文件跟著功能走**：每加一塊能力，對應更新部署/維運文件。

## 8. 給想用 AI 做真實專案的人——可複製的 Checklist

- [ ] 先寫**企劃/規格**再開工，讓 AI 有方向（哪怕一頁也好）
- [ ] 要求 AI**分階段**，每段做完**暫停**讓你看
- [ ] 商業/不可逆決策**自己拍板**，別讓 AI 猜
- [ ] 要求 AI **每步驗證**（測試/build/截圖），不接受「應該可以」
- [ ] 即時/事實資料**要求查證**，警覺 AI 的知識邊界
- [ ] **安全三件事**：關危險預設、金鑰進 Secret 不進 repo、CORS 收斂
- [ ] 早設 `.gitignore`（產物、`.env`、大檔），避免污染 git 歷史
- [ ] 把「上線」當成一等公民：migration、健康檢查、部署文件一起做
- [ ] 認證類動作（OAuth、GitHub 登入）**你親自做**，AI 只能引導

---

> **結語**：AI 能把「一份企劃」變成「可上線的系統」，但好結果來自**好的協作節奏**——
> 人給清楚的方向與決策、AI 給扎實的實作與誠實的驗證，再加上對安全與邊界的共同警覺。
> 這就是 Vibe Coding 真正的樣子：不是放手不管，而是**有方向的共創**。
